USE CASE ID: 4.5<br/><br/>USE CASE DESCRIPTION:<br/>The user wants to cancel a booked services and if he already paid the service he wants also the money back<br/><br/>ACTORS (GOAL):<br/>The User wants to cancel a booked service<br/>The Application  informs the service-provider about the cancelation<br/>The service-provider cancels the booking<br/>The payment-provider refunds the booking fee<br/><br/>PERFORMANCE GOALS:<br/><br/><br/>PRE CONDITIONS:<br/>User selected the booking detail of the booking to be canceled<br/><br/>POST CONDITIONS:<br/><br/><br/>SCENARIO:<br/>1. User selects change booking and presses cancel<br/>2. The application checks the current booking status<br/>3. If the status is already fulfilled or in progress: Alternate 1<br/>4. The application asks the service-provider for cancelation of the booking<br/>5. If the service-provider does not accept cancelation: Alternate 1<br/>6. The service provider accepts the cancelation <br/>7. If there has to be done a refund: Extension 1<br/>8. The application marks the booking as canceled<br/>9.  The application sends a confirmation of cancelation to the user<br/>10. The application shows a confirmation summary<br/><br/>EXTENSION POINT:<br/>Extension 1:<br/>1. The application requests a refund of an amount the service-provider requested for refund<br/>2. The payment-provider refunds the requested amount<br/>3. The provider changes the payment status to ‘canceled’ if the money is not received yet or to ‘to refund’ if the provider debited already<br/>4. The payment-provider returns the new payment-status to the application<br/><br/>ALTERNATIVE SCENARIO:<br/>The application shows a cancelation summery with rejected cancelation request and the reason<br/><br/>RELATED REQUIREMENTS:<br/><br/><br/>RELATED RULES:<br/><br/><br/>PERFORMANCE MEASURES:<br/><br/><br/>VERSION/HISTORY:<br/>0.1; 19.06.15; LÃ¼cking; added actors<br/>0.2; 01.07.15; Stiffel; Added Scenario<br/>
