USE CASE ID: 4.2<br/><br/>USE CASE DESCRIPTION:<br/>The user books a service at the airport. The booking process and an optional payment have to be done. At the end the user has booked a service and got a receipt.<br/><br/>ACTORS (GOAL):<br/>The user wants to book a service from a service provider.<br/>The service provider wants to get a definite booking of a service and also the payment for it.<br/>The payment provider has the goal to provide a secure payment.<br/><br/>PERFORMANCE GOALS:<br/><br/><br/>PRE CONDITIONS:<br/>The user has an account for the seamless travel platform. <br/>The service list is displayed<br/><br/>POST CONDITIONS:<br/><br/><br/>SCENARIO:<br/>1. The user presses the button to book a service<br/>2. The application loads the first form-page from the service provider, who offers the service, extends this form with already known data and asks the user to check the filled and fill out the missing fields. <br/>3. The user checks and fills out the form and presses next step.<br/>4. The application pre-validates the form and sends it to the service-provider to receive the next booking-step<br/>5. Number 2. To 4. Are repeated until the last form is sent.<br/>6. The application shows a booking summary to be checked by the user<br/>7. The user confirms the summary<br/>8. Extension-point for optional payment (see below)<br/>9. The application request a fixed booking from the service-provider<br/>10. The application provider confirms the booking<br/>11. The application sends the confirmation to the user and displays it<br/>12. The user leaves the booking process<br/><br/>EXTENSION POINT:<br/>1. The application sends the payment data to the payment-provider<br/>2. The application displays the payment-provider’s form<br/>3. The user process the payment-process without any communication with the system.<br/>4. At the end of payment process the payment-provider send a payment-confirmation to the system<br/>5. The application rechecks the confirmation by asking the payment-provider for the payment-status<br/>6. The payment-provider confirms the request<br/><br/>ALTERNATIVE SCENARIO:<br/><br/><br/>RELATED REQUIREMENTS:<br/>PCI compliance and privacy requirements are dealt with.<br/>Payment provider may be interchangeable (linked to service).<br/><br/>RELATED RULES:<br/><br/><br/>PERFORMANCE MEASURES:<br/><br/><br/>VERSION/HISTORY:<br/>0.1; 19.06.15; LÃ¼cking; added actors<br/>0.2; 01.07.15; Stiffel; Added Scenario<br/>

Steps:
1. The user presses the button to book a service<br/>
2. The application loads the first form-page from the service provider, who offers the service, extends this form with already known data and asks the user to check the filled and fill out the missing fields.<br/>
3. The user checks and fills out the form and presses next step.<br/>
4. The application pre-validates the form and sends it to the service-provider to receive the next booking-step<br/>
5. Repeat steps 2, 3 and 4 until the last form is sent..<br/>
6. The application shows a booking summary to be checked by the user<br/>
7. The user confirms the summary<br/>
8. Extension-point for optional payment (see below)<br/>
9. The application request a fixed booking from the service-provider<br/>
10. The application provider confirms the booking<br/>
11. The application sends the confirmation to the user and displays it<br/>
12. The user leaves the booking process<br/>
